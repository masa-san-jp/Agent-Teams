# スキル: 相互レビュー (Peer Review)

## 目的
別チームのエージェントの直近の意思決定・成果物・ルールに対して、外部の視点でフィードバックを返し合い、サイロ化と思考の硬直化を防ぐ。「自分では気づけない盲点」を相互に補正する。

## いつ使うか
- 隔週金曜（自己レビューと同日）
- ローテーション表（下記）に従って自分が「レビューア」になっている週
- スポット依頼: 重要意思決定の前に他チームの観点が欲しいとき

## ローテーション表

`spec.json` の `review_cadence.peer_rotation` を正本とする。週番号は ISO 週番号 mod N（N = ローテーション週数）で決定する。

例（6 エージェント・5 週で全 15 ペアを網羅）：

| 週 | レビューペア（双方向） |
|----|----------------------|
| W1 | A ↔ B, C ↔ D, E ↔ F |
| W2 | A ↔ C, B ↔ E, D ↔ F |
| W3 | A ↔ D, B ↔ F, C ↔ E |
| W4 | A ↔ E, B ↔ D, C ↔ F |
| W5 | A ↔ F, B ↔ C, D ↔ E |

各組織のエージェント数に応じてローテーションを設計してください（`config-templates/spec.json.template` 参照）。

## 入力
- レビュー対象の `agents/{相手}/logs/current.jsonl` 直近 2 週間分
- `agents/{相手}/rules.json` 最新版
- `Agent-team/reviews/self/{相手}/` の最新自己レビュー（あれば）
- 自分が見えるレビュイーのアウトプット（チーム間データフローで届いたもの）

## 手順

### 1. 事前確認
1. ローテーション表で今週の自分の担当（誰をレビューするか）を確認する
2. 過去の自分発フィードバック（`reviews/peer/*_{自分}_to_{相手}.md`）を読み返し、未消化の指摘がないか確認する
3. レビュイーの自己レビュー最新版があれば必ず先に読む（重複指摘を避ける）

### 2. 4 観点で評価
評価は「自チームの目的から見て、相手の動きが何を助け／何を妨げているか」を起点に行う。

| 観点 | 問い |
|------|------|
| 共通原則の遵守 | `spec.json.common_principles` が守られているか |
| データフローの質 | 自チームへ来るデータ・依頼の粒度・期日・根拠は十分か |
| 判断の透明性 | rules.json と実際の判断にズレはないか・判断根拠は再現可能か |
| 改善ループ | 同じ issue を繰り返していないか・skills 化のチャンスを逃していないか |

### 3. レビュー対象の制約（やらないこと）
- 個別判断の正否を断定しない（事実と観点を提示し、結論はレビュイー&オーナーに委ねる）
- 自チームの方針を相手に押し付けない（共通原則違反の指摘に限る）
- 機密性の高い詳細（顧客名・契約金額の生データ）はレポートに転記しない（参照のみ）

### 4. レポート出力
`Agent-team/reviews/peer/YYYY-MM-DD_{reviewer}_to_{reviewee}.md` を以下で作成する。

```markdown
# 相互レビュー: {reviewer} → {reviewee} — YYYY-MM-DD

## 対象期間
YYYY-MM-DD 〜 YYYY-MM-DD（2 週間）

## サマリー（3 行以内）
{相手の動きの要約と、最重要フィードバック 1 点}

## 4 観点フィードバック
| 観点 | 観察 | 提案 | 緊急度 |
|------|------|------|--------|
| 共通原則の遵守 | … | … | 高/中/低 |
| データフローの質 | … | … | 高/中/低 |
| 判断の透明性 | … | … | 高/中/低 |
| 改善ループ | … | … | 高/中/低 |

## 質問（レビュイーへ）
- Q1: {判断根拠が読み取れなかった点}
- Q2: …

## 受領前回レビューのフォローアップ
- 前回 {YYYY-MM-DD} のフィードバック {N} 件のうち、対応確認できたのは {M} 件
```

### 5. レビュイー側の対応プロトコル
受領したレビューに対して、レビュイーは次回自己レビュー時までに次のいずれかで応答する。

- **採用**: `rules.json` / `CLAUDE.md` / `skills/` を更新し、自己レビューレポートに「peer review {file path} の {提案} を採用」と記載
- **保留**: 採用判断にオーナー確認が必要な場合、`open_questions` に追加し次回オーナー対話で提示
- **却下**: 採用しない場合は理由を自己レビューレポートに明記（黙殺は禁止）

却下が 3 件続く同観点の指摘は、両エージェントとオーナーで 30 分のすり合わせを設ける。

### 6. ログ・コミット
1. レビューア側: `logs/current.jsonl` に `task: "peer_review"`、`summary` にレビュイー名と指摘件数を追記
2. レビュイー側: 次回自己レビューで対応状況を必ず記録
3. `git_push` でコミット&push する

## アンチパターン
- 「特に問題ありません」とだけ書く（4 観点全てを埋め、最低 1 つは具体的提案を出す）
- 細かい体裁・表記揺れだけ指摘して本質に踏み込まない
- 相手のドメイン専門に踏み込みすぎる（自分の専門観点から見える論点に絞る）
- レビュイーが採用/却下/保留のいずれの応答もしない（黙殺）

## 注意事項
- レポートは 100〜250 行を目安にする
- 緊急度「高」が出た場合は、隔週サイクルを待たずオーナーへ即共有する
- `reviews/peer/` は git 管理対象（gitignore で除外しない）
- 機密データ（個人名・金額生データ）は転記せずパス参照のみとする
