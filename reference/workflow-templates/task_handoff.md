# Skill: タスク引き渡し（task_handoff）

## 目的
オーナーまたは各エージェントが別チームにタスクを渡す際のプロトコル。
受け取り側が「何を・なぜ・いつまでに・どう判定するか」を即理解できる引き継ぎを保証する。

## チーム選択ガイド（オーナー用）

各組織の `spec.json.agents[]` を正本とする。例：

| タスク内容 | 担当チーム |
|----------|----------|
| 技術設計・実装 | `<engineering-like>` |
| 財務試算・ROI 評価 | `<finance-like>` |
| 競合・市場調査 | `<research-like>` |
| プロセス改善・調達 | `<ops-like>` |
| ブランド・対外発信 | `<brand-like>` |

迷う場合は `delegate-suggest` スキルで提案を受ける。

## 引き渡しフォーマット

`agents/{受け取り側}/input/YYYY-MM-DD_<topic>.md` に以下を保存：

```markdown
# タスク引き渡し — YYYY-MM-DD

## From
{送り手 agent or オーナー}

## To
{受け取り側 agent}

## 目的（1〜2 行）
{なぜこのタスクが必要か}

## 期限
YYYY-MM-DD まで

## 入力データ
- {ファイル or 参照先}
- {ファイル or 参照先}

## 期待する出力
- 形式: {markdown / json / spreadsheet / ...}
- 保存先: {agents/{受け取り側}/output/...}
- 受入条件:
  - {基準 1}
  - {基準 2}

## 制約
- {使ってよいツール・参照していいデータの範囲}
- {NG 事項}

## 関連コンテキスト
- 過去の関連タスク: {tasks/done/T-XXX}
- 参考ログ: {logs/...}
```

## 受け取り側の手順

1. `input/` に新規ファイルがないか `startup` スキルで確認
2. 上記フォーマットを読み、目的・期限・受入条件を理解
3. 不明点があれば即 `open_questions` に追加し送り手に確認
4. タスク完了後、`output/` に保存し `input/<file>` を `done/` に移動
5. 送り手に通知（`peer-inbox` の `done` で送信）

## 制約

- **入力ファイル必須**: 口頭・チャットだけの依頼は受け付けない（再現性のため）
- **期限明示必須**: 「なるはや」は受け付けない
- **受入条件 3 個以内**: 多すぎると判定不能になる
- **機密データ**: ファイル本文に転記せず、パス参照のみ

## アンチパターン

- 目的を書かない（「これやって」だけ）
- 期限を書かない
- 出力形式・保存先を指定しない
- 入力データの所在を曖昧にする（「あの資料」）
