# DESIGN-PHILOSOPHY.md — 設計判断の根拠

このドキュメントは「**なぜこういう設計なのか**」を説明します。
受け取り側組織が自前で構築する際、各設計判断を**そのまま採用するか・組み替えるか**を判断できるよう、
背景と代替案、トレードオフを明示します。

`README.md` は「何があるか」、`ARCHITECTURE.md` は「どう組まれているか」、
このドキュメントは「**なぜそう組んだか**」を扱います。

---

## 0. 全体方針

**設計上の最優先事項は「人間が監督できる範囲に AI を留める」**ことです。
スピード・コスト効率・自動化の網羅性は二次的な目標として扱います。

具体的には：
- 不可逆操作（削除・外部送信）は提案止まり、実行は人間
- すべての判断・出力は監査可能なログを残す
- メタエージェントの観測は「自分の出力を再帰観測しない」など暴走防止策つき
- 機密情報の検出に Sentinel を組み合わせる（人間の見落としを補完）

---

## 1. なぜ 4 層モデルなのか

### 採用：ルール → エージェント → スキル → テンプレート

**理由**：

1. **責務の分離**：各層が単一の責任を持つ。「ルール = 守るべきこと」「エージェント = 主体」「スキル = 道具」「テンプレート = 手順」を混ぜない
2. **変更頻度の階層化**：下位ほど安定（ルールはほぼ不変）、上位ほど戦術的（テンプレは頻繁に追加）。混ぜると下位のリファクタが上位を巻き込む
3. **入れ替え容易性**：単一層の中で項目を追加・差し替えしても他層に影響しない

**代替案と却下理由**：

- **平坦な構成**（rules.md と skills.md だけ）：規模が小さいうちは可だが、エージェント数が増えると判別不能になる
- **エージェント単位のサイロ構造**（agents/cfo/{rules, skills, ...}）：横断的な知見が共有されず、重複が累積する

**組織で採用する際の検討事項**：

- 業務エージェントが 1〜2 個なら 4 層は重い。3 層（ルール / エージェント / 道具）でもよい
- 逆にエージェント数が 10 を超えると、エージェント層を**ロール（pure 業務役割）と人格（UI 識別）**に分離した 5 層が必要になる可能性

---

## 2. なぜメタエージェント 4 種なのか

### 採用：reviewer / scout / lab / janitor の 4 役割

**理由**：

1. **継続的な品質維持の自動化**：人間が毎週レビューする時間はないが、AI なら毎日できる
2. **責務の完全分離**：各メタエージェントは**他メタエージェントの責務に踏み込まない**
   - reviewer: 内部標準化（外部参照・削除実行禁止）
   - scout: 外部観測（リポ書き換え禁止）
   - lab: スキル横展開（ルール変更提案禁止 / Reviewer の責務）
   - janitor: クルフト整理（物理削除禁止）
3. **暴走時の被害局所化**：単一エージェントが暴走しても、他の責務には影響しない

**なぜこの 4 種か**：

| 役割 | 解決する問題 |
|------|------------|
| reviewer | ルール・規約のドリフト（時間とともに守られなくなる） |
| scout | 外部のベストプラクティスを取り込めない（井の中の蛙） |
| lab | 重複処理・未使用スキルが累積する |
| janitor | リポジトリのクルフト（不要物）が放置される |

これら 4 つは**継続運用での持病**で、人間が片手間で対応できるサイクルを超えている。

**代替案と却下理由**：

- **単一の "supervisor" エージェント**：責務が広すぎて判断ミス時の影響が大きい。プロンプトも長大化する
- **週次 1 回の手動レビュー**：人間が忘れる。優先度が下がって 1 ヶ月放置されがち
- **クラウド cron + 専用インフラ**：コスト効率が悪い。SessionStart hook + Claude 内部の Agent ツールで十分

**組織で採用する際の検討事項**：

- まず reviewer + janitor の 2 種から始める（lab/scout は外部資源・スキル蓄積が増えてから）
- 各メタエージェントの観測対象は組織のサイズに合わせて調整（過剰観測は finding 過多を生む）

## 3. なぜ「提案のみ」原則なのか

### 採用：すべての破壊的・不可逆操作は提案止まり、実行は人間

**理由**：

1. **AI の判断が間違っていた時の救済可能性**：削除されたファイルは戻らない、送信されたメッセージは取り消せない
2. **権限の最小化**：「観測してよい」と「実行してよい」を分離する
3. **監査可能性**：何が提案され、何が承認/却下されたかが追跡できる

**具体的な制約**：

- janitor は `rm` `git rm` を**生成しない**（コマンド文字列としても出さない）
- scout は外部 URL 参照のみ可、リポジトリ書き換え不可
- reviewer は rules.json への変更**提案**のみ、自動適用しない

**代替案と却下理由**：

- **AI に削除も任せる**：誤削除のリスクが大きすぎる。「24h 前のファイルは古い」と判定されて重要文書が消える事故が起きる
- **承認制・自動実行**：「24h 経って承認なければ自動実行」など。これも事故時の救済が困難

**組織で採用する際の検討事項**：

- 自社が「AI に物理削除を任せたい」場合は、サンドボックス環境（テストリポ）+ 復元可能なバックアップ前提で限定許可
- 完全自動運用は CI/CD パイプライン側で別途実装

## 4. なぜ agent-call / delegate-suggest / peer-inbox の 3 種か

### 採用：用途の異なる 3 つの協調パターン

**理由**：

| パターン | 用途 | 実装 |
|---------|------|------|
| `agent-call` | 1 ターンの委譲（短時間で答えがほしい） | subprocess `claude --print` |
| `delegate-suggest` | 「他エージェントに振った方がよい」と気づいたとき提案 | プロンプト内ロジック |
| `peer-inbox` | 別ターミナルで動く長時間タスクへの依頼 | ファイルベース inbox |

**なぜ 3 つに分けるか**：

- agent-call の同期呼び出しでは、長時間タスクはタイムアウトする
- peer-inbox の非同期メッセージでは、即答が必要な軽い質問にはオーバーヘッド過大
- delegate-suggest がないと、エージェントが自分の専門外でも頑張って答えてしまう（質低下）

**代替案と却下理由**：

- **MCP サーバーで統一**：MCP は外部リソースアクセスに強いが、エージェント間呼び出しには重い
- **共有メモリ**：競合状態の管理が必要。複数 PC・複数ユーザーへの拡張が困難
- **Slack/Discord 統合**：外部 SaaS 依存。ログが社外に出る

**組織で採用する際の検討事項**：

- 並行作業がない（1 人で 1 ターミナル）なら peer-inbox は不要
- エージェント数が 1〜2 なら delegate-suggest は不要（誰に振るか自明）
- agent-call だけは早期に入れる価値が高い

## 5. なぜ Codex（ChatGPT）連携を入れたのか

### 採用：第二意見・並列調査・定型コード生成の 3 用途で ChatGPT を補助的に活用

**理由**：

1. **モデルの多様性で死角を補完**：同じモデルだけだと同じパターンの間違いを繰り返す
2. **不可逆判断の前の検証**：本番デプロイ・契約決定など、戻れない判断には別モデルの確認が効く
3. **並列調査の合成**：Claude+ChatGPT で別視点を得て統合する

**入れない選択肢**：

- 機密情報を含むタスクには絶対使わない（OpenAI に送信されるため）
- 深いコンテキストが必要な作業（Claude 自身がやる方が早い）

**代替案と却下理由**：

- **Claude のみで完結**：単一モデルへの過度な依存。バイアスが補正されない
- **複数 LLM プロバイダ統合フレームワーク**（LangChain 等）：オーバーヘッド大、設定複雑

**組織で採用する際の検討事項**：

- ChatGPT サブスクリプションがない組織は飛ばしてよい
- セキュリティポリシー上、外部 SaaS への送信が禁止なら不採用
- このパッケージでは Codex の実装本体は含めず、必要な組織が独自構築する設計（`tools/README.md` 参照）

## 6. なぜプレースホルダ（`{{...}}`）方式の抽象化なのか

### 採用：5 種の標準プレースホルダで組織固有値を分離

**理由**：

1. **明示性**：`{{ORG_REPO_PATH}}` のように一目で「ここは組織値」と分かる
2. **検出可能性**：grep で機械的に発見できる（`check-placeholders.sh` で漏れ検出）
3. **置換の容易性**：sed で一括置換可能
4. **テンプレートとして JSON 互換**：プレースホルダを残したままでも JSON パーサが通る（注意：value 位置のみ）

**代替案と却下理由**：

- **環境変数（`$ORG_REPO_PATH`）**：シェル展開のタイミング依存で複雑、コメント内では効かない
- **フロントマター + テンプレートエンジン**（Jinja2 等）：依存追加。Bash + Python だけで完結したい
- **絶対パスをハードコード**：そもそも配布できない

**組織で採用する際の検討事項**：

- プレースホルダ規約は **README.md と INSTALL.md のセクション 0 で正本管理**（自動チェックの allowlist と一致させる）
- 独自プレースホルダを追加するなら `check-placeholders.sh` の `ALLOWED_PLACEHOLDERS` に同時追加

## 7. なぜ SessionStart hook + 24h 経過判定なのか

### 採用：Claude Code セッション開始時に overdue 検出 → marker 作成 → 自動起動

**理由**：

1. **コスト効率**：人がそもそも Claude Code を起動しない日は、メタエージェントも走らない（自然な間引き）
2. **インフラ不要**：クラウド cron や外部 scheduler を使わない
3. **可視性**：起動時に「24h 以上未起動: reviewer, lab」と表示されるため、人間がリズムを認識できる

**24h を選んだ根拠**：

- 1 時間：頻度過多。findings の dedup で 90% スキップになり無駄
- 1 日：自然な単位。1 営業日に 1 回の品質チェックで十分
- 1 週間：スパースすぎて issue が累積した時に発見が遅れる

**代替案と却下理由**：

- **完全な cron**：Mac のスリープ・出張・休日で動かない。インフラ管理が増える
- **PR トリガー**：PR を出さない日は走らない。日次の整合性チェックには不向き
- **時刻指定（毎日 9:00 等）**：Mac の電源状態に依存

**組織で採用する際の検討事項**：

- リモートチームで起動時刻がバラバラなら、社内サーバーで cron を併用してもよい
- 24h 閾値はリポジトリの活動量で調整（高頻度なら 12h、低頻度なら 48h）

## 8. なぜ業務エージェントを含めず、テンプレートのみ提供するか

### 採用：業務ドメインエージェントは各組織で独自構築、本パッケージはテンプレ（`config-templates/`）のみ提供

**理由**：

1. **業務ドメインは組織固有**：「CFO エージェント」と一口に言っても、各社の財務プロセスは違う。汎用エージェントは何にも使えない
2. **ドメイン知識は時間とともに育てるもの**：既製品を渡すより、各組織が自分達のログ・判断パターンから育てる方が定着する
3. **配布物の中立性**：特定業界に偏ったエージェントを含めると、別業界では使いにくい

**何を含めて何を含めないか**：

- 含める：エージェントの**設計テンプレ**（`CLAUDE.md.template`, `rules.json.template`）
- 含めない：実際の業務エージェント（cfo / hr / engineering など）

**組織で採用する際の検討事項**：

- まず 1〜2 個の業務エージェントから始める（最大の課題領域から）
- メタエージェントは最初から全 4 種入れる方が、後付けより楽

## 9. 設計を支える「失敗から得た教訓」

このパッケージの設計は、**実運用での事故・苦労**から逆算しています。
具体的に何を防ぐためにどの仕組みを入れたかを記録します。

| 起きた問題 | 入れた対策 |
|----------|----------|
| 個人メールアドレスがコミットされかけた | Sensitivity Sentinel（pre-write hook） |
| 同じ議論が 3 ヶ月後にまた起きた | dev-logs.md と decisions/ で判断履歴を保存 |
| メタエージェントが既出 finding を再提出 | dedup（過去 4 週の同一 finding はスキップ） |
| pending_updates が 30 件溜まり判断不能 | backpressure（5 件超で新規停止） |
| 削除提案が 1 件もなく unused が累積 | 「4 週連続 deletion=0 で警告」メトリクス |
| 受け取り側のプレースホルダ置換漏れ | check-placeholders.sh `--strict` モード |
| Codex タイムアウトで全文レビュー不可 | セクション分割で実施・ドキュメント化 |
| エージェント名 hardcode で他者導入不能 | agents/ ディレクトリスキャン化 |

## 10. このパッケージが解決していない問題

正直に言うと、以下は未解決です。各組織で別途検討してください。

1. **新メンバーのオンボーディング**：このパッケージ自体の使い方を覚えるコストは小さくない。社内ドキュメント or 動画教材は別途必要
2. **複数組織間の知見共有**：各組織がサイロで運用するため、他社のベストプラクティスは Scout 経由でしか入らない
3. **リアルタイム監視**：すべて非同期。「今この瞬間にエラーが起きた」を即時通知する仕組みは別途必要
4. **マルチユーザー間の競合**：peer-inbox は file-based なので、同一 inbox を複数人が同時編集すると競合する（運用で回避）
5. **コスト管理**：agent-call の発火回数や Codex 利用量を集計するダッシュボードはない（メタログは出力するが集計は別途）

## 11. 採用判断のチートシート

「うちの組織で何を入れるべきか？」のクイックガイド：

| 規模・状況 | 最小構成 | 拡張順 |
|----------|---------|--------|
| 1 人・小規模 | rules/ + 業務エージェント 1 個 | janitor → reviewer → lab → scout |
| チーム 3〜5 人 | 上記 + agent-call + meta-agents 4 種 | peer-inbox → codex |
| チーム 10 人以上 | フル装備 | 独自スキル横展開・組織内 sync 強化 |

**最初から全部入れない方がよい**：理解できないものを動かすと、issue が起きた時に対処不能になる。

## 12. このドキュメントの更新方針

設計思想は変わります。新たな失敗・成功から学んだら、本ドキュメントの該当セクションに追記してください。
履歴は git log で追跡。大きな方針転換は冒頭のサマリーボックス（追加予定）で告知します。
