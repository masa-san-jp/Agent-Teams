# SHARING.md — 所属組織内での共有

このパッケージは、まず**配布側オーナーの所属組織内**で共有する想定です。
外部組織への一般公開ではないため、SaaS や公開リポは不要です。

---

## 共有方法

組織内コミュニケーションチャネル（**Slack / Confluence / 社内メール**）でこのフォルダ全体を案内すれば足ります。

### Slack / Teams での共有メッセージ例

```
Claude Code を組織で運用するためのベストプラクティス集を共有します。

📂 場所: <Drive リンク or 社内パス>
📚 中身: 設計思想 + 実装例（全 43 ファイル、~150KB）

【最初に読むもの】
1. README.md — 全体の位置づけ
2. DESIGN-PHILOSOPHY.md — なぜそう組んだか（最重要）
3. ARCHITECTURE.md — 4 層モデル

【すぐ動かしたい場合】
4. INSTALL.md の手順を Claude Code に渡せば自動導入されます

【採用判断】
全部入れる必要はありません。DESIGN-PHILOSOPHY.md § 11 の「採用判断のチートシート」で
組織規模・既存資産に合わせて取捨選択してください。

【質問】
読んでも不明な点は <連絡先> までお願いします。
```

## 共有経路の選択肢

| 経路 | 向いてる場面 |
|------|------------|
| Drive 共有リンク（閲覧者権限） | 外部メンバーやパートナー組織を含む |
| 社内 Confluence / Notion ページに転記 | 検索性・更新追跡を重視 |
| 社内 Git に push | エンジニア組織で git 文化が根付いている |
| Slack に zip 添付 | 1 回限りの共有・小規模 |

オーナーの所属組織の文化に合わせて選んでください。

---

## 更新通知

`reference/` を更新したら、Slack/メール等で「○○を更新しました」と一行通知。
詳細な変更追跡が必要な段階に来たら `CHANGELOG.md` を整備（Phase 4 deploy-010 で予定）。

---

## 共有ログ

組織内で共有した経緯を記録したい場合、以下を記述する場所として活用してください：

| 共有日 | 共有先（チームや個人） | 経路 | 反応・フィードバック |
|--------|---------------------|------|------------------|
| YYYY-MM-DD | （例：エンジニアチーム） | Slack | （例：「meta-agents の責務分離が参考になった」） |

機密性の高い詳細は社内ドキュメントに記載し、ここには概要のみ書く運用を推奨。
